# PRM v0.2 — Implementation Plan: the relationship store, schema builder, and data-floor

*Status: draft for review (2026-06-15). Companion to [`../docs/prm-feature-spec.md`](../docs/prm-feature-spec.md)
(F2/F3, INV-3/4/9/10–13) and [`../docs/roadmap.md`](../docs/roadmap.md) (v0.2). Builds on
[`v0.1-implementation-plan.md`](v0.1-implementation-plan.md). Requirements were extracted in a Grill-Me
session captured at [`../brainstorms/2026-06-15-prm-v0.2-relationships-schema.md`](../brainstorms/2026-06-15-prm-v0.2-relationships-schema.md)
— that file is the decision record this plan formalizes.*

## 1. Goal of v0.2

Give the user a **private, editable, user-shaped relationship store on top of their mirrored contacts**,
and make sure none of it can leak to an external system without an explicit, informed choice. Concretely:

- A single private store, **`relationships.db`** (renamed from `private.db`), holding the existing
  dedup/identity decisions **plus** a user-defined **custom schema** (typed fields), per-contact **field
  values**, **tags**, **notes**, and **contact images**.
- A **schema builder** (the user authors their own typed fields — INV-3, workspace-only) and an in-place
  **contact edit mode** (full CRUD on the relationship overlay, reversible and audited).
- A **data-floor**: every relationship field carries a `disclosure_tier` defaulting to **`private-sealed`**,
  and the MCP surface is structurally unable to emit a sealed field — so new private data is sealed from
  the moment it exists (the v0.2 realization of PNT's `AC-MCP-C`/`PR-7`/`EX-H9`, candidate **AC-PRM-G**).
- Per-field **AI-write tiers** (`review-required` default / `append-only` / `free-write`, INV-10–13) and
  the **`EX-CLOUD-LLM`** consent handler (EX-H2–H7) for cloud MCP clients.

**This realizes** PNT's `F2`/`F3`, advances `INV-3/4/7/9/10–13`, and is the named **demonstrator** for the
upstream data-floor + append-only/free-write ACs (which land in the PNT spec *with* this design, per
CONTRIBUTING — upstream roadmap Tier 1).

### What this plan deliberately holds back
- **Generic (non-image) file attachments** — only contact *images* are in scope now.
- **Groups** (contact-set, many-to-many) — a fast-follow; the upstream `private-data-ops` contract is
  groups-centric, so revisit for demonstrator parity.
- **The config file + platform data dirs** and **source-discovery scanner** — separately-tracked v0.2
  items from the v0.1 plan's "deferred" list; independent of this work, not blocked by it.
- **Outreach / public-data gathering** — v0.3+.

## 2. The core architectural decision (from the grill)

**`shared.db` stays an immutable mirror (INV-2/AC-1). Every user/AI write lands in `relationships.db`,
and the canonical contact remains a *projection*.** The user's "edit mode" is delivered by the same
substrate dedupe already uses:

| CRUD op | Mechanism (all in `relationships.db`) | Status in v0.1 |
| --- | --- | --- |
| **Update** a contact field | `field_resolutions` override (`rule="user"`); projection layers it over the immutable source value | substrate exists (M3); needs UI + endpoint |
| **Create** a user-authored contact | a new `local_records` table; projection unions it in | new |
| **Delete** a contact | a `tombstones` row; projection filters it out | new |
| **Annotate** (tags / notes / custom fields / images) | `field_values` keyed by `contact_id` + `field_id` | new |

This **avoids** the syncing problem a writable `shared.db` would create: a re-mirror can always re-pull a
clean `shared.db`, and edits re-attach by stable id (INV-5/6). Everything is reversible (snapshot ring)
and audited (append-only log) for free.

> **Naming note for attestation.** The file is `relationships.db`; the PNT spec's conceptual term is the
> "Private DB." The `Architecture.md` attestation maps `relationships.db` → the spec's Private-DB surface.

## 3. Data model (`relationships.db` schema v2)

Existing v1 tables are unchanged: `contacts`, `identity_map`, `field_resolutions`, `dedup_decisions`,
`settings`. v2 **adds**:

```sql
-- The user-authored schema. INV-3: written ONLY by the daemon/CLI (workspace), never by MCP.
CREATE TABLE field_definitions (
    field_id        TEXT PRIMARY KEY,        -- stable slug, e.g. "how_we_met"
    label           TEXT NOT NULL,           -- display label
    kind            TEXT NOT NULL,           -- text|long_text|number|date|boolean|single_select|multi_select|url|image
    config          TEXT NOT NULL DEFAULT '{}',  -- JSON: select options [{value,label,description}], number min/max, multi?, …
    class           TEXT NOT NULL DEFAULT 'user', -- 'builtin' (locked) | 'user' (removable)
    ai_write_policy TEXT NOT NULL DEFAULT 'review-required', -- review-required|append-only|free-write (INV-10)
    disclosure_tier TEXT NOT NULL DEFAULT 'private-sealed',  -- private-sealed|private-shareable-on-consent (PR-7)
    required        INTEGER NOT NULL DEFAULT 0,
    position        INTEGER NOT NULL DEFAULT 0,
    created_at      TEXT NOT NULL,
    updated_at      TEXT NOT NULL
);

-- Per-contact values. The value→definition FK is the INV-3/4 structural lock: a value for an
-- undefined field cannot be written. Multi-valued fields (tags, multi_select, image) get >1 row.
CREATE TABLE field_values (
    value_id    TEXT PRIMARY KEY,            -- sha1(contact_id ∥ field_id ∥ value_norm)
    contact_id  TEXT NOT NULL REFERENCES contacts(contact_id) ON DELETE CASCADE,
    field_id    TEXT NOT NULL REFERENCES field_definitions(field_id) ON DELETE CASCADE,
    value       TEXT,                        -- scalar value, or option id, or media content-hash (image)
    value_json  TEXT,                        -- structured payload when needed
    written_by  TEXT NOT NULL,               -- 'manual:user' | 'ai:<session>'
    source      TEXT,                        -- provenance (e.g. 'manual', 'directory-import', export source)
    written_at  TEXT NOT NULL
);
CREATE INDEX idx_field_values_contact ON field_values(contact_id);

-- User-authored contacts (CRUD-Create). Projection unions these with the mirrored set.
CREATE TABLE local_records (
    contact_id  TEXT PRIMARY KEY,            -- minted locally; same id-space as identity_map contact ids
    raw_jcard   TEXT NOT NULL,
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL
);

-- Soft delete (CRUD-Delete). Projection filters these out; source record stays in shared.db (INV-2).
CREATE TABLE tombstones (
    contact_id  TEXT PRIMARY KEY,
    reason      TEXT,
    created_at  TEXT NOT NULL
);

-- SUPERSEDED (R7, 2026-06-16): this table was NOT built. A photo is an ordinary `field_values` row
-- (value = content hash, value_json = {mime,byte_size}, source) — see docs/design-notes/contact-images.md.
-- Kept here only as the original sketch.
-- Content-addressed image references (bytes live on disk, see §4). Sealed by default.
CREATE TABLE media_refs (
    content_hash TEXT NOT NULL,              -- sha256 of the bytes = the on-disk filename stem
    contact_id   TEXT NOT NULL REFERENCES contacts(contact_id) ON DELETE CASCADE,
    field_id     TEXT NOT NULL REFERENCES field_definitions(field_id) ON DELETE CASCADE,
    mime         TEXT NOT NULL,
    byte_size    INTEGER NOT NULL,
    source       TEXT,                       -- 'export:<src>' | 'directory-import' | 'manual'
    created_at   TEXT NOT NULL,
    PRIMARY KEY (content_hash, contact_id, field_id)
);
```

**Seeded built-in fields** (`class='builtin'`, non-removable without a code change): `tags`
(`multi_select`, options = a managed vocabulary with descriptions), `notes` (`long_text`), `photo`
(`image`, single — the avatar). Built-ins are inserted by the v2 migration and re-asserted idempotently on
ensure-schema.

**Projection extension.** `core/projection.py` composes `field_values` + `local_records` + `tombstones`
into the canonical contact: relationship fields are appended (each tagged with its `disclosure_tier`),
local records unioned, tombstoned contacts filtered. The same projection feeds the workspace *and* the MCP
surface — the MCP surface then applies the disclosure filter (§5).

## 4. Contact images — content-addressed media store

> **Decision of record:** [`../docs/design-notes/contact-images.md`](../docs/design-notes/contact-images.md)
> (storage **Option B**, locked 2026-06-16). It refines this section: references live in `field_values`
> (no `media_refs` table); the three ingress paths are tiered (R7 manual upload shipped; R7b Takeout
> auto-match at ingest; R7c guided matcher); uploads downscale client-side; `--raw` includes `media/`.
> The text below is the original sketch.

Bytes live on disk under the PRM home; `relationships.db` only holds references (`media_refs`).

- **Layout:** `<home>/media/<hh>/<sha256>.<ext>` where `hh` = first 2 hex chars of the hash (git-style
  256-way shard). The filename *is* the content hash → distinctive names, automatic **dedup**, **integrity
  check** (re-hash), **immutability** (never edited, only added/removed), trivially **viewable/debuggable**.
- **Why not BLOBs:** the snapshot ring copies the whole `relationships.db` before every apply (×20) —
  image BLOBs would bloat and slow every snapshot. Hash-named files are immutable, so they need no
  per-snapshot copy and the DB stays small.
- **`core/media.py`** (new): `put(bytes) → content_hash` (hash, shard, write-if-absent), `path(hash)`,
  `exists(hash)`, `gc(referenced_hashes)` (orphan sweep), `verify()` (re-hash check for `prm doctor`).
- **Three ingress paths:** (1) **export-borne** — decode the base64 `PHOTO` already inline in
  `shared.db`'s `raw_jcard` into the media store *for display only* (raw stays inline, lossless); (2)
  **directory import** — a `prm import-images <dir>` path matching files to contacts (filename/email/slug
  heuristic with a manual-confirm fallback; heuristic finalized in the design note); (3) **manual upload**
  in edit mode (daemon endpoint hashes + stores + links).
- **Privacy + backup:** images are `private-sealed` → never served over MCP, local-only (INV-1).
  `prm export --raw` MUST include `media/` to stay a lossless round-trip; `prm doctor` reports orphans
  (file without ref, ref without file).

Finer points (single avatar vs multiple images; the directory-matching heuristic) → a dedicated
[`docs/design-notes/contact-images.md`](../docs/design-notes/) written with PR-7 below.

## 5. The data-floor and the MCP seal

**Phase A (ships with the mechanical work) — the minimal seal, safe-by-construction:**
- `field_definitions.disclosure_tier` defaults to `private-sealed`.
- `mcp_servers/tools.py` read bodies (`get_contact`, `list_contacts`, `get_provenance`, `search_contacts`)
  **filter out every field whose `disclosure_tier == 'private-sealed'`** before returning. So relationship
  data **cannot cross the MCP surface** from the instant it exists. This is the first real increment of
  `AC-MCP-C`.
- A test asserts the MCP bodies never emit a sealed field while the daemon/workspace does (the structural
  INV-3-style guarantee, mirrored on the read side).

**Phase B (later PRs) — the full data-floor + consent handler:**
- A workspace control to set a field's `disclosure_tier` (`sealed ↔ shareable-on-consent`) → the
  user-curated **shareable projection**; the cloud-facing read path queries a view that *physically*
  selects only `private-shareable-on-consent` columns (enforced at the query layer, not the UI).
- A static **`disclosure-tier` lint** (clean/dirty fixtures + selftest) + a **dynamic egress probe** in the
  conformance flow — no cloud-facing tool output references a sealed field.
- The **`EX-CLOUD-LLM`** handler: a consent gate before a cloud client first reads shareable private data
  (EX-H2–H5), a persistent "not a PNA right now" banner while active, a return-to-PNA-mode control, and a
  runtime active-set explainer (EX-H7 fail-closed). Down-payment exists today in `mcp_servers/consent.py`.
- **AI-write to values** (`append-only` tier): the MCP private surface gains write access to field
  **values** only (never definitions — INV-3/4, the absent-tool + FK lock), append-only + provenance-
  stamped + per-run quota + audit (INV-12/13). `review-required` stays the default.

## 6. Migration (`private.db` → `relationships.db`)

The rename touches ~65 `private_db` references across ~28 files. It is its own **foundation PR** done
before any new table lands:

- Rename `PrmHome.private_db` → `PrmHome.relationships_db` (+ add `PrmHome.media_dir`); rename

  update every reference in `cli/`, `core/`, `daemon/`, `mcp_servers/`, tests, and docs.
- **Auto detect-and-migrate:** on first v0.2 run, if `private.db` exists and `relationships.db` does not,
  snapshot it, then rename `private.db`(+`-wal`/`-shm`) → `relationships.db`. No manual step. PRAGMA
  `user_version` stays v1 here; the v2 table-add migration runs separately (idempotent ensure-schema bumps
  v1→v2, seeding built-ins).
- A test pins: an existing v0.1 `private.db` is migrated and all v1 behavior is preserved.

## 8. Build order (PR-sized milestones)

Phase 0 is the foundation; Phase A is the usable mechanical system (the bulk of "today"); Phase B is the
safeguards. PRs within a phase are mostly sequential on the spine (R1→R2) then parallelizable.

**Phase 0 — foundation**
- **R0** Rename `private.db`→`relationships.db` + auto-migration + `media_dir` on `PrmHome`. Pure refactor;
  all existing tests green under the new name. *(No behavior change.)*

**Phase A — the mechanical relationship store (safe-by-construction)**
- **R1** Schema substrate: `field_definitions` + `field_values` tables (v2 migration), seeded built-ins
  (tags/notes/photo), `core/schema.py` (define/list/validate; builtin-locked; the value→definition FK
  lock). Tests: schema CRUD, builtin immutability, FK rejects undefined-field value (INV-3/4).
- **R2** Value CRUD + projection extension + **the seal**: projection composes `field_values`; daemon
  get/set/clear value endpoints (lock + snapshot + audit); MCP read bodies filter `private-sealed` (first
  `AC-MCP-C` increment). Tests: sealed fields never cross MCP; daemon does; reversible/audited writes.
- **R3** Contact **Edit mode** UI: edit core fields via overrides, set custom values. Workspace + daemon.
- **R4** **Tags**: managed vocabulary (name+description), assign/unassign, tag-management modal + inline
  create, tags into FTS.
- **R5** **Notes**: built-in long-text per contact, searchable. *(May fold into R3/R4.)*
- **R6** **Schema builder** UI (the "Schema" nav item): full user-field CRUD with kind/config/policy/tier.
- **R7** **Contact images** *(done — "core + show imports" slice)*: `core/media.py` content-addressed
  store, `image` field kind avatar render (list + detail), decode the imported `PHOTO` for display,
  manual upload/remove endpoint in Edit mode, `prm doctor` media-check. Single avatar per contact; a
  photo is stored as a `field_values` row (the separate `media_refs` table was folded into it — no schema
  bump), so it inherits projection / Undo / the MCP seal unchanged. + `docs/design-notes/contact-images.md`.
  **Storage model locked to *Option B*** (2026-06-16, see `docs/design-notes/contact-images.md`): photo
  bytes live as files in the media store, the DBs stay lean (a 5,000-photo store ≈ ~10 MB DB + ~30 MB
  files vs. a ~50 MB inline `shared.db`). **R7b *(done)*:** Google Takeout photos auto-matched at ingest —
  sidecar `.jpg` indexed globally by `FN`, attached on an unambiguous single-image match; bytes in the media
  store, a `prm-media:<hash>` ref in `shared.db` (lean read path); matched photo supersedes a hosted URL
  (INV-1); `--raw` bundles the referenced media (lossless). Coverage is export-dependent (~6% on the sampled
  export — most photos are on archival label-folder contacts the `All Contacts` ingest skips). *Still to
  build:* client-side downscale-on-upload (no image dep); a workspace "Download all (.zip)"; a **fuller
  Takeout import** (archival label folders). **R7c** a guided one-at-a-time visual matcher for loose photo
  folders (filename only *suggests*). Tier-0 polish: make the avatar placeholder itself the upload click target.
- **R8** **Create / Delete** contacts: `local_records` + `tombstones`, projection union/filter, New-contact
  + delete UI.

**Phase B — the safeguards (later sessions)**
- **R9** Disclosure-tier management UI + projection-bound cloud read view + static `disclosure-tier` lint
  (fixtures + selftest). Realizes `PR-7`/`AC-MCP-C` fully.
- **R10** `EX-CLOUD-LLM` consent gate + "not a PNA" banner + return-to-PNA-mode + active-set explainer
  (EX-H2–H7).
- **R11** AI-write-to-values (`append-only` tier) over MCP: values-only, provenance-stamped, quota'd,
  audited (INV-4/12/13). + dynamic egress probe in the conformance flow.
- **R12** Conformance/attestation: update `docs/Architecture.md` (AC-PRM-G data-floor; AC-PRM-E/F write
  tiers; INV-10–13), regenerate `docs/conformance/`, and open the **demonstrator-gated upstream PNT PR**
  landing `AC-MCP-C`/`PR-7`/`EX-H9` + the disclosure-tier lint.

Realistic "today": **R0 → R2** (rename + migration + schema substrate + value CRUD + the seal) is the
spine and the highest-value, safe stopping point; **R3–R7** (edit mode, tags, notes, schema builder,
images) as far as time allows. Phase B is its own session(s).

## 9. Conformance, invariants, and upstream

- **Invariants advanced/added:** INV-2 preserved (shared.db immutable); INV-3 (schema authored only in
  workspace — structural: no MCP definition tool); INV-4 (AI writes values not definitions — R11); INV-7
  (provenance on relationship values); INV-9 (relationships.db durable; wiped only by explicit reset);
  INV-10–13 (write tiers/quota/audit — Phase B); new **disclosure tier** (`PR-7`/AC-MCP-C/candidate
  AC-PRM-G).
- **Upstream is demonstrator-gated:** PRM ships + attests first; the PNT spec PR (AC-MCP-C/PR-7/EX-H9 +
  lint) follows once R9–R12 are green (per CONTRIBUTING; upstream roadmap Tier 1). No upstream edits today.
- **`pyproject.toml` dependency surface:** unchanged — all of this is stdlib (`sqlite3`, `hashlib`,
  `json`, `http.server`). Images need no library (hash + file write). Keep the tiny-deps invariant.
