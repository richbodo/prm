# PRM v0.1 — Implementation Plan

*Status: draft for review (2026-06-02). Companion to [`../docs/prm-feature-spec.md`](../docs/prm-feature-spec.md)
and [`../docs/roadmap.md`](../docs/roadmap.md). This plan implements **v0.1 only**.*

## 1. Goal of v0.1

Stand up the smallest honest slice of the PRM: **import contacts from files, view/search them,
and let an AI deduplicate them through a propose → review → apply flow.** v0.1 is the first
realization of the PNT *Personal Relationship Manager* use case, the draft **AC-PRM-B** multi-source
dedup contract, and the proposed **AC-PRM-E/F** safe-AI-write model (review-required tier only).

**In scope:** CLI ingester; a single **PRM home** data directory + path resolver
(`--data-dir`/`PRM_HOME`, repo-local `./prm-data/` default — demo mode runs entirely inside the
repo); `prm init [--demo]` (the demo home is *seeded from the synthetic fixtures*, never committed as
binary DBs); import **preview + interactive source-confirm** layered over a pure, non-interactive
`ingest()`; local daemon + read/search workspace + proposal-review UI; MCP read tools + propose-only
dedup tools; the dedup prompt; the private store; snapshot ring + append-only audit log; opt-in
non-destructive re-import.

**Out of scope (later versions):** private overlay (groups/tags/notes), custom schema, CardDAV live
mirror, public-data gathering, outreach, append-only/free-write AI tiers, per-vendor API connectors,
the responder app. **Also deferred:** the config *file* + platform data dirs (XDG / macOS / Windows);
the Downloads + ingestion-dir **source-discovery** scanner (a v0.2 library shared by the CLI and the
MCP ingestion surface); any installer. (No version-control backend — the proposal/audit substrate is
native; see §4.)

## 2. Flavor (axis picks for v0.1)

| Axis            | Pick |
| --------------- | ---- |
| distribution    | `never-distributed-single-user` [†](#distribution-verifiability) |
| storage         | `native-sqlite-via-filesystem` (triggers AC-PRM-C single-instance file-lock) |
| ingestion       | `multi-source-merge-with-dedup` (triggers AC-PRM-B) — file import only in v0.1 |
| workspace shell | `vanilla-js-spa` served by a small Python-stdlib local daemon |
| comms           | none in v0.1 |
| mcp-exposure    | `shared-only` + a propose-only dedup surface (see §6) |

<a id="distribution-verifiability"></a>**† Distribution — actual posture vs. the formal pick.**
`never-distributed-single-user` is the only fitting pick today, and it is accurate: PRM is built by
the user themselves, with no per-user delivery story. But it *understates* the trust posture. PRM is
the **independently-verifiable** end of the distribution spectrum — a "home-cooked meal": a friend
goes to the source repo, **builds it themselves, and runs the conformance / evaluate flow to confirm
it is a compliant PNA before trusting it.** The current axis can't express that, collapsing it with
opaque-binary delivery. This is a known gap in the toolkit's distribution axis, not a PRM decision to
revisit — tracked in [`../docs/roadmap.md`](../docs/roadmap.md) (Parallel track), PRM issue
[#8](https://github.com/richbodo/prm/issues/8), and upstream
[personal_network_toolkit#39](https://github.com/richbodo/personal_network_toolkit/issues/39). PRM is
the candidate demonstrator if the axis splits; the **installer** (deferred) is what forces the
question.

## 3. Architecture overview

Four cooperating parts over files on disk:

- **CLI ingester** — the only writer of the raw Shared DB. Run by the user (`prm import …`).
- **Local daemon** — Python stdlib HTTP server; serves the workspace and a read API; the **only
  writer of canonical private data** (applies approved proposals).
- **Workspace (vanilla-JS SPA)** — search/view + proposal-review UI.
- **MCP server** — read-only over contacts; **propose-only** over the private store (stages
  changesets, never commits).

Concurrency: native SQLite with WAL; a single-instance advisory **file-lock** per AC-PRM-C guards
the canonical writers (ingester at import time, daemon at apply time) so they never race. The MCP
server and workspace open contacts **read-only**.

## 4. Data model

Two SQLite files (AC-1 two-store split), plus a text/JSON proposal+audit substrate.

**`shared.db` — raw mirrored records (read-only at runtime; written only by the ingester, INV-2):**
- `source_records(source_record_id PK, source, source_uid, ingested_at, raw_jcard TEXT)` — one row
  per record per source, stored as canonical jCard JSON.
- `field_provenance(source_record_id, field, value, observed_at)` — per-field provenance (AC-17,
  INV-7).
- `contacts_fts` — FTS5 over name/email/org/notes-from-source for search.

**`private.db` — the private store (durable; survives re-import, INV-5/6/9):**
- `contacts(contact_id PK, created_at)` — the stable canonical contact (INV-5).
- `identity_map(source_record_id, contact_id)` — which raw records belong to which canonical
  contact. **This is what dedup writes (to `private.db`, never `shared.db`).** Re-import re-attaches by stable `source_uid`, so decisions
  survive.
- `field_resolutions(contact_id, field, chosen_value, chosen_source, rule, status, resolved_by, resolved_at)`
  — the chosen value for a single-valued field when sources disagree. `rule` is the survivorship basis
  (`source_priority` | `most_complete` | `most_recent` | `user`); `status` is `applied` | `needs_review`.
  A manual field **edit** is the same row with `rule="user"`, so this table is also field-level
  CRUD-**Update** (see [`../docs/design-notes/dedupe-design.md`](../docs/design-notes/dedupe-design.md)).
- `dedup_decisions(pair_key PK, decision, decided_at)` — persisted "not a duplicate" rejections (and
  confirmed merges), keyed on a stable pair/cluster key so a dismissed candidate never resurfaces on
  re-detect or re-import.
- `settings(key PK, value)` — incl. `source_priority` (the reconcile order — config, not hardcoded);
  `PRAGMA user_version` (schema version for the AC-4 handshake).

*(Deferred to a post-M3 slice on this same substrate: `local_records` for user-**created** contacts and
a contact **tombstone** for **delete** — the Create/Delete half of CRUD; Create with v0.2's private overlay.)*

The canonical contact a user sees is a **projection**: `identity_map` ⨝ `source_records` ⨝
`field_resolutions`, with per-field provenance attached.

**Proposal + audit substrate (text/JSON, implemented natively — no VCS):**
- `proposals/<id>.json` — a staged **changeset**: the unit both flows emit and the review UI applies.
  `created_by` is the *only* semantic difference between the manual and AI authors (`"manual:<user>"`
  vs `"ai:local-dedup"`). Operations: `merge {source_record_ids, into, confidence, signals}`,
  `resolve_field {contact_id, field, chosen_value, chosen_source, rule}`, `flag_conflict {contact_id,
  field, candidates, status:"needs_review"}`. `merge`/`resolve_field` record inverse ops
  (`split`/`unresolve_field`) for audit symmetry; v0.1 Undo is snapshot-restore (below), so inverses are
  logged but not yet an apply path.
- `audit.log.jsonl` — append-only record of every applied operation (INV-12).
- `snapshots/private-<iso>.db` — pre-apply snapshot ring of `private.db` (AC-9, INV-12); ISO-sorted
  rotation. **This is v0.1 Undo** — restore the pre-merge snapshot.

## 5. CLI ingester

- Entry: `prm init [--demo]`; `prm import <file|dir>`; `prm status`; `prm reimport <file|dir>`
  (opt-in, see below).
- **Thin CLI, fat library.** The argparse layer only parses args, calls a library function, renders
  output, and sets the exit code. All orchestration lives in a **pure, non-interactive**
  `ingest(paths, *, source, dry_run) → ImportReport` so the same core is driven by the CLI *and* the
  MCP ingestion surface, and tests exercise it without spawning a process or a prompt. Interactive
  bits (the source-confirm, the future discovery pick-list) live only in the CLI and are bypassed by
  `--non-interactive`/`--json`.
- **One PRM home for all stores.** Everything for an instance lives under a single data directory
  (`shared.db`, `private.db`, `proposals/`, `snapshots/`, `audit.log.jsonl`). Location resolves in
  order: `--data-dir` → `PRM_HOME` → (later) a config file's `data_dir` → repo-local `./prm-data/`
  (the demo/dev default — nothing lands outside the repo). `prm init` creates the home; `--demo`
  seeds it by running the ingester over the committed synthetic fixtures (a `demo/` manifest lists
  which sources), giving a realistic ~1000-contact demo with **no external files and no committed
  binary DBs** — and doubling as an end-to-end smoke test.
- **Source is inferred, then confirmed.** `detect_format` gives the *format*; a finer *source* label
  (what lands in `source_records.source` + provenance) is inferred (Apple `PRODID` → `apple_icloud`,
  Takeout zip → `google_takeout`, `Connections.csv` → `linkedin`, …) and shown in a preview
  (detected source + contact count + confidence) for the user to accept or correct. `--source`
  overrides; `--non-interactive` does best-effort + a report and never prompts.
- Format detection + parsers. **Inbound is vCard 3.0-dominant** — Google Takeout, iCloud, and
  Outlook all export 3.0 by default; vCard 4.0 in the wild is rare. Parse server-side with a
  tolerant lexer: **`vobjectx`** (the actively-maintained fork of the now-dormant `vobject`,
  Python ≥3.10) — its value is real-world quirk / line-fold handling, **not** field semantics,
  which we map ourselves. Then **vendor CSV** (stdlib `csv` + a per-vendor column mapping), then
  **Google Takeout** — which is *not a separate format* but a **zip of `.vcf` files** under a
  `Contacts/` tree (stdlib `zipfile` feeding the same vCard path).
- Canonical representation is **jCard (RFC 7095), modeled as our own internal dict.** There is no
  maintained Python jCard library and none is needed: RFC 7095 is a mechanical
  `[name, params, type, value]` mapping we own. So "canonical jCard" is an internal shape, not a
  dependency. (vCard **4.0** *emission* is out of scope for v0.1; if a later version must emit 4.0,
  `pythonvCard4` is the RFC-6350 option — pin it, it is single-maintainer.)
- Assign a stable id via a **fallback chain**, since most real sources carry no `UID`:
  `UID → normalized email → source permalink (e.g. LinkedIn profile URL) → content hash
  (name + org/company + connected-date + source)`. Walk down until a key exists. Real sources land
  all over this chain, and the fixtures exercise each rung: Apple/Google vCards usually omit `UID`
  (email wins); **LinkedIn** has no `UID` and ~98% no email, so the **profile URL** is the key
  (stored as vCard `URL` / jCard `url`); **Facebook** gives only name + connected-date, so the
  **content hash** is the only option.
- **Cross-source merge is separate from id assignment, and confidence-tiered.** Email/phone matches
  are confident; **name+company (LinkedIn) and name-only (Facebook) matches are fuzzy and MUST be
  proposed, never auto-merged** (AC-PRM-B). The content-hash id is deliberately *not* a cross-source
  merge key. Caveat: a content-hash id is fragile — if the name changes (or a source fixes an
  encoding bug, e.g. Facebook's UTF-8 mojibake) the id changes and re-import creates a new contact
  instead of re-attaching; the dedup/review layer compensates.
- Record per-field provenance (source + timestamp).
- Output validation: `PRAGMA quick_check` + zero-row guard (IN-2).
- **Re-import is opt-in and non-destructive (INV-6, AC-10, AC-PRM-D):** stage → validate → preview
  what changes / what `identity_map` rows would be orphaned → commit atomically only on confirm.
  Never background-polls.
- **Dependency surface (deliberately tiny).** Beyond the two parsing/transport libraries, v0.1 is
  stdlib: `http.server` (daemon), `sqlite3` + FTS5 (storage + search), `csv`, `zipfile`, `json`,
  `hashlib`. The only non-stdlib runtime deps are **`vobjectx`** (vCard lexing) and the official
  **`mcp`** SDK (§6) — a far lighter stack than the legacy `prt` ingester this replaces, which
  pulled in SQLAlchemy, Textual, and Torch. Pure Python throughout.

## 6. MCP server(s)

- **Shared Data Ops (read-only):** `search_contacts`, `get_contact`, `list_contacts`,
  `get_provenance`, and `find_duplicate_candidates` (server-side blocking/similarity returning
  candidate clusters with signals, so the AI proposes against real candidates rather than scanning
  blind).
- **Dedup Ops (propose-only — no canonical write):** `submit_merge_proposal(changeset) → proposal_id`,
  `list_proposals()`, `get_proposal(id)`. **No tool commits** — enforces INV-11 / AC-PRM-F
  structurally.
- **Consent posture:** read tools return PII, so cloud clients hit AC-MCP-A; **local AI recommended**.
  Document the consent gate; pin per-surface posture (open question).
- Packaging: official `mcp` SDK in an isolated venv; imports app logic only via pure helpers
  (mirrors fellows's `mcp_servers/` discipline).

## 7. Dedup prompt

Ship a `dedup.md` instruction set telling the AI to: pull candidate clusters; ask the user good
clarifying questions; build a **conservative** changeset; `submit_merge_proposal`; then direct the
user back to the workspace to review and apply. The prompt is the v0.1 substitute for a hand-built
dedup wizard.

## 8. Workspace (vanilla-JS SPA via daemon)

- **Search** (FTS5) + **browse list** + **contact detail** (merged projection + per-field
  provenance + contributing sources).
- **Proposal review UI** — list pending proposals; render each as a PR-style diff (what merges,
  which fields resolve); **approve all / approve some / edit / reject**; on approve, the daemon
  snapshots `private.db`, applies atomically, writes the audit log, and shows an impact/orphan
  preview first.
- PNT Debug contract, single-user-sized: build label (AC-15), a `--diag`/`?diag` state dump, an
  always-reachable reset/escape (AC-6), boot phase marks. Sink/maintainer-mailbox may be empty
  (DB-8).

## 9. Conformance & attestation

Write `docs/Architecture.md` as the specialization-and-conformance layer (fellows is the model):
declare Toolkit-Version + flavor + an **AC attestation table** (Realization / Verification / Status)
covering AC-1, AC-4, AC-6, AC-7, AC-9, AC-10, AC-15, AC-17, AC-PRM-B, AC-PRM-C, AC-PRM-D, AC-MCP-A,
and the **proposed** AC-PRM-E/F. Mark not-applicable ACs (comms ACs, distribution/PWA ACs) with
reasons.

## 10. Testing

- Fixtures: a synthetic, diffable **Google Takeout corpus** under `tests/fixtures/google_takeout/`
  exercises the real-world quirks — vCard 3.0, `itemN.` property grouping, folded lines, missing
  `UID` (forces the INV-5 fallback), unicode, embedded + external photos, intra-export duplicates,
  and the legacy flat layout. A stdlib `build.py` zips the unpacked trees into Takeout-shaped
  `.zip`s. Real exports are never committed; fixtures are generated/synthetic. (This is the project's
  first PR.)
- Unit: parsers/normalizer (vCard/CSV/Takeout → jCard), stable-id assignment, provenance capture.
- DB: schema bootstrap, two-store isolation (private writes never touch `shared.db`), re-import
  stage/validate/swap + orphan preview, snapshot rotation, audit-log append.
- MCP: read tools; `submit_merge_proposal` stages but never commits (the structural INV-11 test).
- E2E: import → search → AI proposes merges → review → apply → re-import preserves merge decisions
  (INV-5/6).

## 11. Build order (milestones)

- **M0** Data model, SQL DDL, snapshot + audit + file-lock substrate, version handshake.
  **Gate the build on FTS5 being compiled into the target `sqlite3`** (stdlib; usually present on
  macOS / modern CPython, but search depends on it — check `sqlite3.connect(':memory:').execute("PRAGMA compile_options")` or a probe `CREATE VIRTUAL TABLE … USING fts5`).
- **M1** CLI ingester, in order: (a) parser layer — vCard + Takeout → canonical jCard + provenance
  + stable id *(done)*; (b) CLI skeleton — `cli/config.py` data-dir resolver, pure `ingest()` +
  `ImportReport`, `prm init [--demo]`, `prm import --dry-run` with the source preview/confirm; (c)
  the `shared.db` load path so `import` persists; (d) `prm status`. (Then the CSV parser:
  LinkedIn + Google CSV.)
- **M2** Daemon + workspace search/view (read path only). *(done — stdlib `http.server` daemon
  (`daemon/server.py`, pure `route()` + socket glue, 127.0.0.1-only) over a read-only `shared.db`
  facade (`list_records`/`get_record`); a vanilla-JS SPA (`workspace/`) with search + browse + contact
  detail incl. per-field provenance; launched by `prm serve`. The §8 PNT Debug-contract polish — `?diag`
  state dump, AC-15 build label, boot-phase marks — rides with the M6 attestation pass; AC-6 escape is
  the SPA `reset`.)*
- **M3** Dedupe — the deterministic/manual flow, end to end (no AI/MCP). Rationale + locked decisions:
  [`../docs/design-notes/dedupe-design.md`](../docs/design-notes/dedupe-design.md). Sub-slices:
  - **M3a** `core/private_db.py` + schema + the **projection** (`identity_map ⨝ source_records ⨝
    field_resolutions`); import seeds `identity_map` 1:1; the workspace now renders canonical contacts.
  - **M3b** `core/candidates.py` — stdlib detection (normalize → block → tier → guarded cluster). This
    *is* `find_duplicate_candidates` (§6), shared with M4's MCP surface.
  - **M3c** `core/proposals.py` + apply path under the file-lock (snapshot → run ops →
    `field_resolutions`/`identity_map` writes → append audit), completing M0's snapshot+audit substrate;
    `dedup_decisions`.
  - **M3d** workspace **Duplicates** queue + PR-style reconcile UI (visual design from a dedicated
    frontend-design session — menu, front-page refresh, the Duplicates surface); daemon
    `GET /api/candidates` / `POST /api/merge`. One-at-a-time, neutral. Field-level **Update** falls out here.
  - **M3e** Undo via snapshot-restore + a Merge-history view.
  - **M3f** **Bulk approve** — a workspace mode that merges a whole group of duplicates (by tier / AI
    author) in one pass: select groups → flip-through spot-check with **inline** conflict resolution →
    one click. Applied as a **single combined changeset** (`core.apply.apply_batch` → the multi-op
    changeset run in one transaction), so the batch is atomic and a single Undo reverses it. Daemon
    `POST /api/merge-batch`; `GET /api/candidates` enriched with per-cluster `conflicts` + survivor
    `into` (`core.projection.clusters_meta`). Rationale: [`../docs/design-notes/dedupe-design.md`](../docs/design-notes/dedupe-design.md) Decision 5.
- **M4** MCP shared-data-ops (read) + dedup-ops (propose-only) + `dedup.md`.
- **M5** Opt-in non-destructive re-import with preview; wire snapshot ring + audit log end-to-end.
- **M6** `Architecture.md` attestation + full test pass + conformance review.

## 12. Proposed file layout

```
prm/
  cli/        prm_import.py · ingest.py · report.py · config.py · normalize.py · provenance.py
              jcard.py · identity.py · parsers/{vcard,csv,takeout}.py
  core/       shared_db.py · private_db.py · proposals.py · audit.py · snapshots.py · schema/*.sql
  daemon/     server.py            # stdlib HTTP; serves workspace + read/apply API
  workspace/  index.html · app.js · sw? (none) · styles   # search/view + proposal review
  mcp_servers/ shared_data_ops.py · dedup_ops.py · tools.py · prompts/dedup.md   # not `mcp/` — the SDK's package *is* `mcp`
  demo/       manifest.json        # which synthetic fixtures seed `prm init --demo` (JSON: 3.10, no tomllib)
  docs/       Architecture.md      # conformance/attestation layer
  tests/      unit/ · db/ · mcp/ · e2e/
```

`prm_import.py` is argparse + dispatch only; `ingest.py` is the pure orchestrator; `config.py`
resolves the PRM home. Discovery (`prm scan` / Downloads) is a v0.2 addition and will land as
`cli/discover.py`, shared with the MCP ingestion surface.
