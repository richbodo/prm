# PRM v0.2 — Custom Relationship Data Schemas: Brainstorm / Discovery Notes
Date: 2026-06-15 · Goal: Extract every requirement/decision needed to write a concrete v0.2 implementation plan (`plans/`) for a user-defined relationship-data store ("relationships DB"), a schema builder, a CRUD UI, and the later-stage MCP/data-floor/consent safeguards. Build as much as can be done today.

## Summary / key decisions
_(running synthesis, updated as the grill proceeds)_

- **[Q1] Store = rename `private.db` → `relationships.db` (ONE unified private store).** Holds the existing dedup/identity tables (`identity_map`, `field_resolutions`, `dedup_decisions`, `settings`) **plus** the new relationship overlay + custom-schema tables. Implications: (a) real, in-file FK enforcement for both `contact_id` and value→field-definition; (b) needs a **migration** for existing v0.1 `private.db` files (rename/detect-and-migrate, version-handshake bump); (c) churns every code/doc reference to `private.db`/`home.private_db`; (d) the **data-floor disclosure tiers are THE sealing mechanism** (relationship values share the store the projection/MCP already read — no store-absence guarantee), which matches the spec's PR-7 model. Attestation still maps the file to the spec's conceptual "Private DB."
- **[Q2] Tags + notes ship now, as seeded built-in fields inside the generic schema** (`relationships.db`), keyed by `contact_id`. Two field classes: **`builtin`** (tags, notes — non-removable without a code change) vs **`user`** (full create/remove). Tags are a **managed vocabulary** (name + description) assigned many-to-many, with a management modal + inline create; tags feed search.
- **[Q2] User wants editable contacts ("edit mode").** **CRITICAL: the values they edit must NOT be written to `shared.db` (that violates INV-2/AC-1).** PRM's existing design delivers the same UX via the private-store override + projection: **Update** = `field_resolutions` (rule="user"), **Create** = a new `local_records` store, **Delete** = a tombstone — all in `relationships.db`, with `shared.db` staying an immutable mirror. This also *avoids* the "syncing problems" the user worried about (a re-mirror can't clobber edits; they re-attach by stable id, INV-5/6). **Pending user confirmation.**
- **[Q2] "Today" = as much of v0.2 as possible across several sequenced, PR-sized milestones; multi-session OK.**
- **[Q3] Contact "edit mode" via the override model** (CONFIRMED): Update→`field_resolutions`(rule="user"), Create→`local_records`, Delete→tombstone — all in `relationships.db`; `shared.db` stays immutable (INV-2). The projection composes them.
- **[Q4] Field kinds = full set:** text · long-text · number · date · boolean · single-select · multi-select · url · **image** (added Q7). Every field carries `class` (builtin/user), `ai_write_policy` (default review-required), `disclosure_tier` (default private-sealed); config JSON (select options carry `{value,label,description}`).
- **[Q5] UI = new "Schema" nav item** (define fields, INV-3 workspace-only) + **in-place contact Edit mode** (fields/tags/notes/custom/image) + **tag-management modal** (name+description, inline create) + New-contact + delete. Convert the "Coming soon" nav placeholders.
- **[Q6] Minimal seal ships with the mechanical phase** (disclosure_tier default sealed + MCP excludes sealed = first increment of AC-MCP-C); **full data-floor + EX-CLOUD-LLM consent gate = Phase 2 PRs.**
- **[Q7] Contact images IN now**, stored as **content-addressed files on disk** (`media/<hh>/<sha256>.<ext>`), `image` field kind, sealed by default. Generic non-image attachments deferred.
- **Defaults baked in (unless changed):** auto detect-and-migrate `private.db`→`relationships.db` (snapshot+rename+version bump); PNT spec ACs land as a **demonstrator-gated upstream follow**, not today; **groups deferred**; full doc + test updates per repo discipline.

## Context already established (from code + roadmap reads, not from the user)
- **v0.1 stores:** `shared.db` (raw mirror, INV-2, ingester-only writer) + `private.db` (dedup/identity: `identity_map`, `field_resolutions`, `dedup_decisions`, `settings`). Canonical contact = projection over both, joined by `contact_id`.
- **PNT spec term** for the private store is **"Private DB"** (fellows + `contracts/private-db.schema.sql`, `mcp-private-data-ops.schema.json`). User's working name is "relationships DB" — naming tension to resolve.
- **v0.2 scope (both roadmaps):** private overlay (groups/tags/notes) → user-defined custom schema (typed fields, workspace-only authorship, INV-3) → per-field **AI-write tiers** (review-required default / append-only / free-write, INV-10–13, AC-PRM-E/F) → **data-floor** (per-field disclosure tier, default `private-sealed`; projection-bound cloud MCP; AC-MCP-C/PR-7/EX-H9 = candidate AC-PRM-G) → **EX-CLOUD-LLM** consent handler (EX-H2–H7).
- **Upstream gating:** the data-floor + append-only/free-write ACs land in the PNT spec **with** the PRM v0.2 demonstrator (per CONTRIBUTING). PRM v0.2 is the named demonstrator (upstream roadmap Tier 1).
- **User's stated ordering:** dedupe-first (identity must be settled before relationship data attaches by `contact_id`); build the *mechanical* schema-builder + value CRUD first, **then** the leak-prevention safeguards.
- **PRM storage axis** is `native-sqlite-via-filesystem` → it does NOT inherit the PWA `CST-*` ceilings.

## Q&A log

### Q1 — Store layout & naming
- Asked: Where should the new relationship data live (schema defs + values + groups/tags/notes)?
- Captured: **Rename `private.db` → `relationships.db`** — one unified private store for everything (dedup decisions + relationship overlay + custom schema). User chose this over my recommended "new 3rd store" and over "extend private.db." Accepts the migration + churn cost in exchange for one clean store with enforced FKs.
- Flags: Migration path for existing v0.1 `private.db` → owner: this plan (decide rename-on-first-run vs detect-and-migrate).

### Q2 — Feature scope (tags/notes, contact editing) + "today" ambition
- Asked: generic schema only, or also dedicated groups/tags/notes?
- Captured (rich answer; several decisions):
  - **Tags + notes are IN, seeded up front for usability.** Principle: "users must be able to *use* something to gain benefit" — ship a working system fast even if a pure rebuild-it-yourself purist would omit them. They live **inside the generic schema** of `relationships.db`.
  - **Built-in fields are non-removable.** The user should NOT be able to delete the seeded tags/notes fields "without changing the code of the program itself." → seeded/built-in field_definitions are locked; **user-defined** custom fields ARE removable. (Two classes of field: `builtin` (locked) vs `user` (full CRUD).)
  - "**There is no extant definition of a relationships.db schema, so make our best guess.**"
  - **Contact editing / "edit mode" wanted.** User pictures an **edit mode per contact**: edit the contact's info, attach **tags**, **notes**, and **attachments**. User SAID "when you edit the contact info, it gets saved into the shared DB" and acknowledged "this causes all kinds of syncing problems down the road" but "if the app is going to be used, it must be usable" — and explicitly asked me to **check whether the architectural constraints prohibit this.**  → **See my assessment below: writing shared.db violates INV-2/AC-1, and PRM already has the mechanism that delivers this exact UX without that — the `field_resolutions` override + projection (field-level Update), with Create→`local_records` and Delete→tombstone. NEEDS USER CONFIRMATION (next Q).**
  - **Tags model:** a **tags table** where "a tag can have a tag [name] and a description"; assigned tags are stored in the private store (`relationships.db`) **keyed by contact_id** (many-to-many). Needs a **tag-management modal** (edit name + description) and **inline "create a new tag"** when assigning and the wanted tag isn't in the picker.
  - **Search:** "you just add tags to search" — tag (and likely note) values feed search/FTS.
  - **Scope/ambition:** expect to get **a lot of v0.2 done this session across several PRs**; multi-session is fine (user may break for sleep). Plan should be **PR-sized, sequenced milestones**; execute as many as possible today.
- Flags:
  - **Attachments per contact** = NEW concept (binary/file storage + upload UI). → owner: this plan — propose deferring or a minimal version; confirm with user.
  - Groups (contact-set, many-to-many) NOT raised this round → treat as deferred fast-follow unless user wants it.
  - Tag unification (dedicated tags tables vs "tags = a built-in multi-select field whose options are a managed vocabulary with descriptions") → my recommended design, confirm with user.

### Q3 — Edit storage (INV-2 check the user requested)
- Asked: deliver "edit mode" via the private-store override + projection (keep shared.db immutable), or write edits into shared.db?
- Captured: **CONFIRMED — Override model.** shared.db stays an immutable mirror (INV-2/AC-1 preserved). Update → `field_resolutions` (rule="user"); Create → `local_records` store; Delete → tombstone; all in `relationships.db`; the projection composes them. Reversible + audited + re-import-safe; avoids the sync problem the user worried about. The user's "saved into shared.db" instinct is superseded by this (same UX, invisible layering).
- Flags: none.

### Q4 — Field kinds + per-field metadata
- Asked: which field kinds should the schema builder support?
- Captured: **Full set — text · long-text · number · date · boolean · single-select · multi-select · url.** Tags = built-in multi-select; Notes = built-in long-text. Every `field_definition` also carries: **class** (`builtin` locked vs `user` removable), **ai_write_policy** (default `review-required`, INV-10), **disclosure_tier** (default `private-sealed`, PR-7), plus key/label/config(JSON: select options, number min/max, etc.)/required/order/timestamps. Metadata columns exist from day one; their **enforcement** is the later safeguards phase.
- Flags: select-option vocabulary needs to carry a description (per the tag model) → config JSON holds options as `{value,label,description}`.

### Q5 — UI layout
- Asked: how to organize the v0.2 UI for schema definition + contact editing?
- Captured: **Nav item + in-place edit.** New top-level **"Schema"** nav item (define/edit custom fields; INV-3 workspace-only authorship). **Contacts detail panel gains an "Edit" mode**: edit fields (→ overrides), assign tags (multi-select + **tag-management modal** with inline "create new tag"), write notes, set custom-field values. **"New contact"** button → `local_records`; **delete** → tombstone. Convert the existing "Coming soon" nav placeholders into the real surfaces.
- Flags: none.

### Q6 — Leak-prevention phasing (the interim seal)
- Asked: how to split leak-prevention, given the existing MCP read tools would otherwise expose new relationship data immediately?
- Captured: **Minimal seal now, full data-floor later.** Mechanical phase ships a **`disclosure_tier` column (default `private-sealed`) + MCP tool bodies structurally exclude sealed fields from ALL output** — so new relationship data cannot cross the MCP surface from the moment it exists (also the first real increment of AC-MCP-C). **Phase 2 (own PRs):** user-curated shareable projection, EX-CLOUD-LLM consent gate (EX-H2–H7), "not a PNA" banner + return-to-PNA-mode, disclosure-tier **lint** + dynamic **egress probe**, AC-PRM-G / AC-MCP-C attestation, upstream PNT spec landing (demonstrator-gated).
- Flags: none — this is the agreed safe-by-construction posture (honors sealed-by-default + "fail loudly").

### Q7 — Attachments → narrowed to CONTACT IMAGES, in scope now
- Asked: attachments in/out for this v0.2 push?
- Captured (user interrupted to redirect): **Contact IMAGES are wanted now ("right away or in very short order") — implement them in the custom schema this push.** Generic file attachments stay deferred; the concrete need is images.
  - **Sources of images:** (a) some contact exports carry them (vCard `PHOTO`, embedded base64 or external URI — PRM already has `takeout-photos` / `takeout-photos-external` fixtures); (b) the user has images from **directory exports** ("not just SIS systems") — i.e. folders of images to match to contacts; (c) manual upload in edit mode.
  - User's instinct: "a folder of JPEGs with distinctive filenames"; explicitly invited me to **research/propose a better method** that is **easy to maintain, reliable, easy to debug**.
  - **My recommendation (to confirm): content-addressed image files on disk** under the PRM home (`media/<hh>/<sha256>.<ext>`, git-style 2-char shard), referenced from `relationships.db`; modeled as a new **`image` field kind**. Rationale below in the chat + to be finalized in a design note.
  - **Why NOT BLOB-in-SQLite:** the snapshot ring (`core/snapshots.py`) copies the *entire* relationships.db before *every* apply (×20 ring) — image BLOBs would bloat/slow every snapshot. Content-addressed files are immutable (hash = name) so they need no per-snapshot copy, dedup automatically, are integrity-checkable, and are directly viewable/debuggable (the PNA "tool-readable / files on disk" ethos).
- Flags:
  - Single vs multi image per contact (recommend: built-in single "photo" avatar field + allow a multi-valued image field) → confirm in plan.
  - Directory-import matching: how a folder of images maps to contacts (filename convention vs manifest vs manual-assign in edit mode) → design note.
  - How vCard `PHOTO` currently flows through ingest/shared.db (inline base64 in raw_jcard?) — VERIFY in code; decide whether to extract PHOTO into the media store at projection/display time vs keep raw inline. → plan task.
  - Images are `private-sealed` by default; `prm export --raw` (lossless backup) MUST include the `media/` folder; MCP never serves image bytes. → plan.
- **CONFIRMED method: content-addressed files on disk.** `media/<hh>/<sha256>.<ext>` referenced from `relationships.db` via an **`image` field kind**. Dedups, integrity-checkable, immutable, debuggable, keeps the snapshot ring small. Sealed by default; `export --raw` includes `media/`; `prm doctor` media-check reports orphans. Finer points (single vs multi image; directory-matching heuristic) → design note.
- **Verified in code:** vCard `PHOTO` is normalized as a `uri`-typed jCard prop → embedded base64 already lands inline in `shared.db`'s `raw_jcard` (lossless, immutable). Plan decodes it into the media store for *display* (raw stays inline); external/directory images are net-new handling.
